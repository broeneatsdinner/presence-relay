# Operator Interface

The private operational deployment includes a read-only `bin/presence` command
for inspecting accepted events and doorway observations. The implementation is
not copied into this public repository. This document records its verified
behavior with sanitized examples so the operator contract can be reviewed
without publishing runtime data or private deployment details.

## Presentation And Compatibility

Human-readable tabular commands use Unicode box tables. Missing correlated
fields render as an em dash (`—`), and elapsed intervals use compact readable
forms such as `53 sec`, `5 min 57 sec`, `1 hr 02 min 03 sec`, or
`1 day 02 hr 03 min 04 sec`.

The existing JSON compatibility surface remains machine-readable and unchanged.
In JSON mode, an unmatched field is `null`, not an em dash:

```text
bin/presence --json doorway arrivals
bin/presence --json doorway departures
```

The box drawing, em dash, and readable duration are presentation choices. They
do not rewrite stored observations or alter JSON values.

## Arrival Correlation

```text
bin/presence doorway arrivals [--limit N]
```

The arrival view asks a deliberately narrow question: after an observed
home-geofence arrival, what was the first subsequent single doorway-button
observation before the next home arrival?

```text
home-geofence arrival
  -> first subsequent single doorway-button observation
```

For each `events` row with `event = 'arrive'` and `place = 'home'`, the command
builds a window that begins at that arrival timestamp, inclusive, and ends at
the next home-arrival timestamp, exclusive. It then selects the first
`doorway_events` row in that window with `press_type = 'single'`.

The windows are computed before the output limit is applied. A button
observation is not reused, unmatched arrivals remain visible, and results are
shown newest-first. When timestamps are equal, row ID supplies the deterministic
tiebreaker.

Sanitized public transcript:

```text
┌────────────┬───────────────────────────┬───────────┬───────────────────────────┬──────────────┐
│ arrival_id │ entered_home_geofence     │ button_id │ pressed_doorway_button    │ elapsed      │
├────────────┼───────────────────────────┼───────────┼───────────────────────────┼──────────────┤
│ 105        │ 2026-01-14T17:42:10-07:00 │ 22        │ 2026-01-14T17:47:31-07:00 │ 5 min 21 sec │
│ 103        │ 2026-01-14T08:18:02-07:00 │ —         │ —                         │ —            │
└────────────┴───────────────────────────┴───────────┴───────────────────────────┴──────────────┘
```

## Departure Correlation

```text
bin/presence doorway departures [--limit N]
```

The departure view asks the complementary question: after a single
doorway-button observation, what was the first subsequent leave-home geofence
event before the next single doorway observation?

```text
single doorway-button observation
  -> first subsequent home-geofence departure
```

For each `doorway_events` row with `press_type = 'single'`, the command builds a
window that begins at the button timestamp, inclusive, and ends at the next
single doorway-button timestamp, exclusive. It then selects the first `events`
row in that window with `event = 'leave'` and `place = 'home'`.

The windows are computed before the output limit is applied. A departure event
is not reused, unmatched doorway observations remain visible, and results are
shown newest-first. When timestamps are equal, row ID supplies the deterministic
tiebreaker.

Sanitized public transcript:

```text
┌───────────┬───────────────────────────┬──────────────┬───────────────────────────┬─────────┐
│ button_id │ pressed_doorway_button    │ departure_id │ left_home_geofence        │ elapsed │
├───────────┼───────────────────────────┼──────────────┼───────────────────────────┼─────────┤
│ 23        │ 2026-01-14T18:03:11-07:00 │ 106          │ 2026-01-14T18:04:04-07:00 │ 53 sec  │
│ 22        │ 2026-01-14T17:47:31-07:00 │ —            │ —                         │ —       │
└───────────┴───────────────────────────┴──────────────┴───────────────────────────┴─────────┘
```

Both commands default to `--limit 10` and accept values from `1` through `500`.
The limit controls returned rows, not the construction of correlation windows.

The same sanitized tables are tracked as presentation transcripts under
[`demo/transcripts/`](../demo/transcripts/README.md). They are documentation
fixtures, not output generated by the public local demo.

## Interpretation Boundary

These commands correlate observations; they do not prove why either event
happened. Four layers remain distinct:

| Layer | What it can say |
| --- | --- |
| Observed doorway event | A single button observation was recorded at a time. |
| Geofence transition | The mobile device reported arriving at or leaving the home geofence. |
| Bounded correlation | Two eligible observations occurred in a deterministic, non-overlapping matching window. |
| Inferred meaning | A possible account of physical movement, which the commands do not assert as fact. |

Unmatched rows are operationally useful evidence. They expose missing,
out-of-window, late, or otherwise unpaired observations instead of manufacturing
a complete narrative. Likewise, non-reuse prevents one event from lending
artificial support to multiple anchors.

Raw insertion remains raw: doorway rows are recorded without arrival,
departure, occupancy, or causal meaning. Correlation happens later through a
read-only operator view, leaving the original doorway and geofence streams
independently inspectable.

## Verification And Publication Boundary

In the private operational repository, focused operator CLI tests, the full test
suite, Python compilation, CLI help checks, and the Git whitespace check passed.
Live arrival output was also checked against the operational database. This is
verification evidence, not a claim that private implementation or live output
is present here.

The transcripts in this repository use synthetic timestamps and invented IDs.
They intentionally omit live rows, private paths, hostnames, account names,
network details, provider details, database locations, and deployment commands.

Confidence scoring, learned windows, ambiguity classification beyond an
unmatched result, occupancy inference, and journey-level interpretation remain
future ideas. They are not implied by the bounded correlation commands.
